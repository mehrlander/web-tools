# markdown in chat — working visually on a phone

A companion to [`github/markdown.md`](github/markdown.md), split by which
renderer you are writing for. That doc is **GitHub's static `.md` renderer**:
what a committed markdown file turns into on github.com. This doc is the
**chat client's renderer** (the provider's app, read mostly on a narrow
mobile screen), which has its own quirks and its own good moves. The two
overlap but are not the same renderer, so a trick that works in one can fail
in the other.

The findings below were established empirically in a chat session, screenshot
by screenshot. They are provider-general enough to travel, but a new client
is worth a quick re-probe.

## The core problem: bullets balloon vertically

A chat renderer gives each markdown list item paragraph-level vertical
margin, and nesting compounds it, so a nested bullet list is mostly
whitespace on a phone. A three-deep tree of bullets can run a full screen for
a handful of entries. This is the single biggest layout tax in chat markdown.

**Tables and fenced code blocks render vertically tight** by comparison: row
height and line height are compact, no per-item margin. So the move, whenever
content has structure that a bullet list would carry, is to reach for a table
(links live, content flows) or a code block (monospace, aligned, but no
links). Prefer the table when the items should be tappable; the code block
when alignment matters more than navigation.

This is the layout half of the surfacing conventions' existing "longer, not
wider" instinct (see the `caption` skill's folder tables and the `tasks`
board): one item per tight row beats packing several into a wide cell, and
both beat a bullet cascade.

## The table cell trims leading whitespace

Inside a markdown table cell the renderer strips leading and trailing
whitespace and collapses internal runs of spaces to one. Tested: **ASCII
space, non-breaking space (U+00A0), and tab are all trimmed** at the start of
a cell. So indentation can never ride on plain blank space; the first visible
thing in a cell defines its left edge.

What *does* survive as a leading indent:

- **A visible glyph:** a box-drawing rail `│`, a filetype icon, a bullet
  mark. It renders, so it holds the column open.
- **An inline code span:** whitespace inside backticks is preserved and
  rendered monospace. So `` `│  ├─ ` `` keeps its exact spacing and aligns
  column-to-column. The catch: a code span cannot contain a link.
- **The braille blank U+2800** (and, less reliably, the fancier Unicode
  spaces: figure U+2007, em U+2003, ideographic U+3000). These are printable
  non-space characters, so the whitespace trim does not touch them. U+2800 is
  the pick: a true blank glyph, even width, never at risk of being reclassed
  as space.

A quick reference, from the probes:

| Technique | Survives leading? | Holds a link? | Aligns |
|---|---|---|---|
| ASCII space / nbsp / tab | no | n/a | n/a |
| braille blank U+2800 | yes | yes (blank, then link) | ~monospace |
| visible rail `│` / icon | yes | yes (glyph, then link) | proportional drift |
| inline code span prefix | yes | no (link goes outside) | monospace, exact |

## The payoff: a file tree in chat

The reason all of this got worked out. A repo tree is the natural case that
wants both hierarchy (indentation) and navigation (links), which is exactly
what bullets do badly and a bare code block cannot do at all. Two formats
came out of it, both one node per table row with a filetype icon anchor and
the name linked to its GitHub blob/tree URL:

**Code-span box art** (the most capable). The `tree`-style connector prefix
goes in backticks so it renders monospace and aligned; the icon and link sit
outside the span:

| Tree |
|---|
| 📁 [.claude](https://github.com/mehrlander/web-tools/tree/main/.claude) |
| `├─ `📁 [hooks](https://github.com/mehrlander/web-tools/tree/main/.claude/hooks) |
| `│  └─ `📄 [build-on-commit.sh](https://github.com/mehrlander/web-tools/blob/main/.claude/hooks/build-on-commit.sh) |
| `└─ `📁 [skills](https://github.com/mehrlander/web-tools/tree/main/.claude/skills) |

**Braille indent + icons** (quieter, best when shallow). Depth is pure
U+2800 indentation, no connectors; the icon carries folder-vs-file:

| Tree |
|---|
| 📁 [.claude](https://github.com/mehrlander/web-tools/tree/main/.claude) |
| ⠀⠀⠀📁 [skills](https://github.com/mehrlander/web-tools/tree/main/.claude/skills) |
| ⠀⠀⠀⠀⠀⠀📄 [SKILL.md](https://github.com/mehrlander/web-tools/blob/main/.claude/skills/caption/SKILL.md) |

Both are generated by [`scripts/build-tree.py`](../scripts/build-tree.py) and
operated by the [`tree`](../.claude/skills/tree/SKILL.md) skill, which owns
the full parameterization (scope, depth, mode, gloss). A plain fenced code
block (aligned box art, no links) stays as the pasteable fallback.

## Rules of thumb

- Structure that would be a nested bullet list → a markdown table.
- Depth/indentation in a table → a surviving character (code span, U+2800,
  or a visible rail), never plain space or tab.
- Tappable node → the link sits outside any code span.
- Alignment matters more than links → a fenced code block.
- New chat client → re-probe the trim and the survivors before trusting
  them; the behavior is renderer-specific.
