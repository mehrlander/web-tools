# Give show-repo the ability to edit a repo's .web-tools.json

show-repo already reads a repo's `.web-tools.json` manifest and can write one field
(the staged-files list, via Save stage). The direction is to let it **edit the
config more generally**, so a repo's landing, pins, and other web-tools settings
can be set from the shell rather than by hand-editing the file.

## First use: migrate the legacy manifest name

The rename from `.show-repo.json` to `.web-tools.json` (PR #222) left a read
fallback so old repos keep working. The clean way to end that fallback is a
one-tap **migrate** action in the shell: when the shell loads config from the
legacy name, offer "Migrate to .web-tools.json," which writes the new file from
the loaded contents. This is the first concrete instance of config editing.

Constraint found while scoping: `gh-store.js` has `save`/`saveBytes` (Contents
API PUT) but no delete. A write-only migrate leaves the old `.show-repo.json`
inert (harmless, since `loadConfig` reads the new name first), but a true rename
needs a delete. Either add a delete helper to the gh layer, or leave old-file
cleanup to a git-capable session.

## Scope notes

- The back-compat itself is a code concern (where the scripts look), not a model
  instruction, and is already tagged `SUNSET(2026-08-15)` so it resurfaces for
  removal. This task is what lets that removal happen cleanly: get repos onto the
  new name via the shell.
- A general config editor (landing, pins, stage, and future fields) is the larger
  goal; the migrate action is the thin first slice that proves the write path.
