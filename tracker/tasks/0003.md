# Singleton fab with toss-render integration

The fab should be a singleton: the second load detects the first and hands its context up rather than mounting a second button. This resolves the toss-render conflict where both the renderer and the rendered page mount their own fab, producing two overlapping buttons with disjoint information.

## What "done" means

- `fab.js` enforces single-instance mounting. A second call contributes its context (loaded scripts, Alpine components) to the existing fab instead of creating a new one.
- The fab's drawer partitions its inventory into two tabs when running inside toss-render: a "Toss" tab (source ref, original URL, rewrite log) and the normal "Page" tab (the inner page's components and scripts, filtered to exclude toss-render plumbing).
- Outside toss-render, behavior is unchanged (single tab, current inventory).

## Design notes

- Toss-render injects the page into the same document (not an iframe in `#gh=` mode), so both fabs share a DOM. The singleton guard is a simple global check.
- The inner page's scripts load asynchronously after injection, so the fab's inventory snapshot must wait for the page's boot to finish before populating the "Page" tab. A `MutationObserver` or debounce after the last script load handles this.
- Toss-render can tag its own scripts with a data attribute so the fab can partition "plumbing" from "payload" without a hardcoded list.
