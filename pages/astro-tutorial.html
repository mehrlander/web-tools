01. What Astro is (and isn't)

Astro is a web framework for content-driven sites. Think marketing pages, docs sites, blogs, portfolios, e-commerce storefronts — anywhere most of the page is HTML and only some bits need to be interactive.

Mechanically: you write .astro components that look like JSX-flavored HTML, plus optional React / Vue / Svelte / Solid / Preact components for the interactive parts. Astro ships zero JavaScript to the browser by default and only hydrates the components you explicitly opt in.

Astro IS
  • • A static-site generator with optional SSR
  • • HTML-first, with a JSX-like component syntax
  • • A multi-framework host (React + Vue + Svelte coexist)
  • • Built on Vite — fast dev, ESM everywhere
  • • Great for content + occasional interactivity
Astro is NOT
  • • A SPA framework (no client-side router by default)
  • • A great fit for highly stateful apps (use Next/Remix/SvelteKit)
  • • Tied to React — JSX-ish, but not React
  • • An app shell with global state baked in
  • • A drop-in replacement for Next.js

02. The mental model: islands & zero-JS by default

The single most important Astro concept is the islands architecture:

  • A page is mostly server-rendered static HTML — no JS framework runtime in sight.
  • You sprinkle islands of interactivity (a search box, a cart icon, a chart) where you actually need them.
  • Each island ships only its own JS, hydrates independently, and doesn't block the rest of the page.

Compare to a typical SPA where the whole page is React/Vue, and even static text gets re-rendered on the client. Astro flips that: HTML is the default, JS is opt-in, per component.

If your page is 95% content and 5% interactive widgets, Astro's defaults will produce a much smaller payload than a SPA framework with the same UI.

03. Getting started

Spin up a project with the CLI:

# Node 18.17+ or 20+
npm create astro@latest my-site
cd my-site
npm run dev      # http://localhost:4321
npm run build    # static output in ./dist

Add a UI framework as an integration when you need interactivity:

npx astro add react        # or vue, svelte, solid, preact
npx astro add tailwind     # official integrations exist for most tooling
npx astro add mdx

04. Project layout 

my-site/
├─ astro.config.mjs     # integrations, output mode, adapters
├─ package.json
├─ public/              # static assets, copied as-is (favicon, robots.txt)
└─ src/
   ├─ pages/            # 🔥 file-based routing lives here
   │  ├─ index.astro      # → /
   │  ├─ about.astro      # → /about
   │  └─ blog/
   │     └─ [slug].astro  # → /blog/:slug  (dynamic route)
   ├─ layouts/          # shared shells (header/footer wrappers)
   ├─ components/       # reusable .astro / .jsx / .vue / .svelte
   ├─ content/          # typed content collections (markdown, mdx, json)
   └─ styles/

Anything in src/pages/ with an .astro, .md, or .mdx extension automatically becomes a route. No router config to write.

05. The .astro file format

An .astro file has two parts: a component script at the top (TypeScript, runs at build time on the server), and a component template below (HTML with JSX-style expressions). 

---
// Component script — runs on the server only
import Card from '../components/Card.astro';
const title = 'Hello, Astro';
const posts = await fetch('https://api.example.com/posts').then(r => r.json());
---

<html>
  <head><title>{title}</title></head>
  <body>
    <h1>{title}</h1>
    <ul>
      {posts.map(p => (
        <li>
          <Card title={p.title} href={`/blog/${p.slug}`} />
        </li>
      ))}
    </ul>

    {posts.length === 0 && <p>No posts yet.</p>}
  </body>
</html>

<style>
  /* scoped to this component automatically */
  h1 { font-size: 2rem; }
</style>

Things worth noticing

  • The frontmatter (--- fences) is normal TypeScript. await at the top level is fine.
  • Expressions in the template use { }, like JSX.
  • HTML attributes are HTML — it's class, not className; for, not htmlFor.
  • <style> tags are scoped to the component by default. <style is:global> opts out.
  • <script> tags in .astro files are processed by Vite, bundled, and shipped to the browser.
  • The script runs once at build time (or per request, in SSR mode). It never runs in the browser.

06. File-based routing & dynamic routes

Static routes are just files. Dynamic routes use bracket syntax and a getStaticPaths() export that tells Astro which paths to generate. 

// src/pages/blog/[slug].astro
---
export async function getStaticPaths() {
  const posts = await fetch('https://cms.example.com/posts').then(r => r.json());
  return posts.map(post => ({
    params: { slug: post.slug },
    props: { post }
  }));
}

const { post } = Astro.props;
---

<article>
  <h1>{post.title}</h1>
  <div set:html={post.bodyHtml} />
</article>

In SSR mode, dynamic routes don't need getStaticPaths — they run per request and read params from Astro.params.

07. Data fetching & content collections

In a .astro file, you can await anything in the frontmatter: hit a database, read the filesystem, call an API. There's no getServerSideProps or loader() ceremony — just write async code.

For local Markdown / MDX content, Astro provides content collections with Zod-validated schemas:

// src/content/config.ts
import { defineCollection, z } from 'astro:content';

const blog = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    pubDate: z.date(),
    draft: z.boolean().default(false),
    tags: z.array(z.string()).optional()
  })
});

export const collections = { blog };
// src/pages/blog/index.astro
---
import { getCollection } from 'astro:content';
const posts = (await getCollection('blog'))
  .filter(p => !p.data.draft)
  .sort((a, b) => +b.data.pubDate - +a.data.pubDate);
---

<ul>
  {posts.map(p => <li><a href={`/blog/${p.slug}`}>{p.data.title}</a></li>)}
</ul>

You get autocompletion and validation errors at build time when frontmatter doesn't match the schema.

08. Adding interactivity (islands)

Drop a React/Vue/Svelte component into the page like any other component. The crucial bit is the client:* directive — without it, the component is rendered to static HTML on the server and ships zero JS. 

---
import Counter from '../components/Counter.jsx';
import Search from '../components/Search.svelte';
---

<!-- Static HTML only — no JS shipped -->
<Counter start={0} />

<!-- Hydrate immediately on load -->
<Counter start={0} client:load />

<!-- Hydrate when the browser is idle -->
<Counter start={0} client:idle />

<!-- Hydrate when scrolled into view -->
<Search client:visible />

<!-- Hydrate only on a media query match -->
<Search client:media="(max-width: 768px)" />

<!-- Render only on the client, never on the server -->
<Counter client:only="react" />
Default to no directive. Reach for client:visible or client:idle before client:load. Most "interactive" UI doesn't need to hydrate before the user can see it.

09. Layouts, slots & props

Components accept props via TypeScript-typed Astro.props, and children via <slot /> (web-component style, not JSX-children style). 

// src/layouts/BaseLayout.astro
---
interface Props { title: string; description?: string; }
const { title, description = '' } = Astro.props;
---

<html lang="en">
  <head>
    <title>{title}</title>
    <meta name="description" content={description} />
    <slot name="head" />
  </head>
  <body>
    <header>...</header>
    <main><slot /></main>     // default slot
    <footer><slot name="footer" /></footer>
  </body>
</html>
// src/pages/about.astro
---
import BaseLayout from '../layouts/BaseLayout.astro';
---

<BaseLayout title="About" description="Who we are">
  <link slot="head" rel="canonical" href="https://example.com/about" />
  <h1>About</h1>
  <p>...</p>
</BaseLayout>

10. Static vs server output

Astro can output static HTML (default), SSR (rendered per request), or a hybrid. The mode is set in astro.config.mjs: 

// astro.config.mjs
import { defineConfig } from 'astro/config';
import node from '@astrojs/node';

export default defineConfig({
  output: 'server',           // 'static' | 'server' | 'hybrid'
  adapter: node({ mode: 'standalone' }),  // or vercel, netlify, cloudflare…
});

In SSR mode, individual pages can opt out of pre-rendering:

---
export const prerender = false;   // this page renders per request

if (Astro.request.method === 'POST') {
  const form = await Astro.request.formData();
  // handle the submission, then redirect
  return Astro.redirect('/thanks');
}
---

You can also expose API routes by putting a .ts file in src/pages/api/ that exports GET, POST, etc. handlers returning a standard Response.

11. What you're responsible for

The framework is intentionally minimal. Here's the line between what Astro hands you and what you wire up yourself.

Astro gives you You provide
File-based routing, dynamic params, redirects Information architecture, URL design, sitemap entries
Build pipeline (Vite), bundling, dev server, HMR Choosing integrations (React, Tailwind, MDX, image, sitemap)
The .astro component model + slots + scoped styles Designing components, prop shapes, layout hierarchy
Content collections + Zod schemas + type-safe queries Writing the content (Markdown / MDX), defining the schemas
Islands & client:* hydration directives Choosing the right directive per component (perf budget)
Image optimization (<Image />), font handling Source images, alt text, content sources
SSR adapters for Node, Vercel, Netlify, Cloudflare, Deno Hosting choice, env vars, deploy config
Form actions, middleware, API endpoints Auth, sessions, DB connections, business logic
Zero default client JS Any global state, client-side routing, store wiring
View Transitions API integration Deciding which routes participate & the actual transitions

Notably absent from the framework: state management, auth, a CSS solution, an ORM, or a blessed UI library. Astro stays out of those choices. You bring (or don't bring) tools like Tailwind, Lucia, Drizzle, nanostores, etc.

12. When to pick Astro

Reach for it when
  • • The site is content-heavy (blog, docs, marketing, landing).
  • • You want SEO + small payload + fast TTFB without fighting the framework.
  • • You want to mix React for one widget and Svelte for another without religion.
  • • Markdown / MDX is a first-class part of the project.
Look elsewhere when
  • • You're building a SPA-style app (dashboard, editor, chat).
  • • You need deep client-side routing and shared state across many views.
  • • Most of the page is interactive and the "content" is a thin shell.
  • • Your team is heavily invested in Next/Remix/SvelteKit conventions.
