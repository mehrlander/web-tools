Demo of what alpine-bundle.js registers on top of Alpine: magic properties, behaviors, and custom directives. The bundle also loads Alpine itself (and @alpinejs/collapse) from unpkg.

Magics

$toast

$toast(icon, msg, cls?, ms?) → push a transient toast. Backed by $store.toasts; the renderer at the top-right of this page reads from that store.

$clip

$clip(text) → copy text via a hidden textarea + execCommand('copy'). Also fires a toast. Works on mobile and inside data: URLs where the modern clipboard API is gated.

$paste

$paste(cb) → focus a hidden textarea, wait for the next paste, call cb(text). Tap the button then paste (long-press → Paste on iOS, ⌘V on desktop).

Behaviors

Dropdown auto-close

The bundle installs a document-level click handler: any open <details class="dropdown"> closes when you click outside it. Inside-click closing is opt-in via data-auto-close — useful for picker menus, omitted for panel-style dropdowns that contain inputs.

Directives

Custom Alpine directives that decorate the host element via dot-modifiers. Three flavors: class shortcuts (mirror helpers in kits/fills.js), reactive renderers, and behavior wrappers.

x-btn — class shortcut

x-btn.xs.primary="count++" → adds btn btn-xs btn-primary and wires the expression as a click handler. Unknown modifiers pass through as raw classes (x-btn.xs.shadow-md).

x-metric — formatter + reactive

x-metric.currency.trend="{ label, val, prev }" → renders a metric card with Intl-formatted value and a trend indicator. Modifiers shape the rendering pipeline: .currency / .percent for formatting, .trend to enable the diff arrow, .invert to swap the favorable direction.

x-action.confirm — behavior wrapper

x-action.confirm="..." → confirmation-gated click. First tap arms (turns red, swaps label to "Are you sure?"); second tap within 3s executes; otherwise reverts. Without .confirm it's a plain click.

x-define — custom-element registration

<template x-define="kebab-tag">…</template> registers a custom element whose instances clone the template content and let Alpine initialize them in place. Host attributes flow in reactively via the $attrs magic. Data factories use the standard Alpine path — x-data inside the template (with Alpine.data('name', fn) for shared logic).

Custom-element registration is global; the directive warns if a <template x-define> is nested inside an x-data subtree, since the scope-implied positioning would be misleading. Templates live at document level.

Sits alongside the other path the repo uses for reusable Alpine pieces: files under alpineComponents/ that call Alpine.data('name', fn) inside alpine:init, mounted by writing <div x-data="name()">…</div>. Same Alpine factory shape; what differs is the surface (custom-element tag vs. x-data attribute) and where the template HTML lives.

Self-contained counter

<template x-define="demo-counter">
  <div x-data="{ count: 0, inc() { this.count++ } }">
    <button class="btn btn-sm" @click="inc()" x-text="`Count: ${count}`"></button>
  </div>
</template>

Reading attributes via $attrs

$attrs is a magic that returns the host element's attributes as a reactive object. Use it inside any Alpine expression in the template — x-data seed values, x-text, :class, etc. 

<template x-define="demo-greeting">
  <div x-data="{ name: $attrs.name ?? 'world' }">
    Hello, <b x-text="name"></b>! (passed start=<span x-text="$attrs.start ?? 0"></span>)
  </div>
</template>

<demo-greeting name="Ada" start="42"></demo-greeting>

Reaching into a component from outside

Each instance's reactive state lives in its inner x-data scope. Alpine.$data(el) returns that scope so other code can read or call into it directly.

Magics, directives, and the dropdown handler all live in alpine-bundle.js. kits/fills.js is the Alpine-free string-helper counterpart.
